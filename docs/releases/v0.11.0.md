# v0.11.0 - Pair compatibility and reroll memory

Release date: 2026-07-14

## Summary

This release makes composed-meal roulette behave like meal planning instead of a cartesian-product picker. Centerpiece and side dishes are now evaluated as a whole meal: bad combinations such as repeated tuna, fish plus fish, competing animal proteins, or redundant ingredient families are rejected or penalized before selection.

Reroll also gains memory. A slot no longer cycles back through alternatives the user has already seen in the current reroll session, and exhaustion is reported explicitly instead of silently looping.

## Added

### Pair compatibility

- Primary ingredient and primary family diagnostics for recipes.
- Scheduler-facing simple-dish semantic roles such as `protein_centerpiece`, `carb_centerpiece`, `vegetable_side`, and `salad_side`.
- Hard rejection rules for shared primary ingredients, duplicate dominant protein, competing animal proteins, repeated dominant families, and invalid side identity.
- Soft pair scoring for positive complementarity, whole-meal balance, duplicate groups, style overlap, and weak combinations.
- Pair-level reason codes and user-facing selection reasons for composed meals.

### Reroll memory

- Per-slot reroll history stored on `meal_plan_items`.
- Combination keys for complete meal packages, including composed centerpiece+side pairs.
- Reroll exclusion for the current and previously seen combinations.
- Explicit exhausted reroll response with a user-facing message.
- Start-over endpoint to clear reroll memory for a slot.
- Reroll history reset on week regeneration, manual assignment, item deletion, and stale week cleanup.

### Structure-first roulette

- Complete mains and composed pairs are now separate candidate classes.
- Weekly structure policy prefers composed pairs below the configured minimum, complete mains at or above the configured maximum, and uses a neutral share inside the target band.
- Default structure target: 4-7 composed meals per week, with a neutral 60% main / 40% composed split.
- Week generation and reroll use the same structure-aware generator, with fallback only when the preferred class has no valid candidates.

### Package-level planning fixes

- Meal swap exchanges the full meal package, including all lines, roles, sources, recipes, reasons, and manual extras.
- Today cards and cook menus handle multi-line meals more robustly.
- Recipe fetches for composed meals tolerate partial failures so one failed dish does not wipe the whole cook menu.

## Changed

- The scheduler scores composed meals as one candidate rather than ranking every main and pair in one raw pool.
- Pair scoring reuses pre-scored centerpiece and side candidates to keep generation interactive with larger simple-dish catalogs.
- Similarity and variety labels use whole-meal labels and vectors for composed assignments.
- Stale reroll cleanup no longer commits inside locked generation/reroll transactions.
- Planning rule defaults include `composed_meals_per_week` and `structure_neutral_share`.

## Data and migration notes

- New migration: `032_reroll_history`.
- Adds `meal_plan_items.reroll_history_json` for backend-side reroll memory.
- Existing meal plans are compatible; reroll history starts empty.

## Upgrade notes

- Run Alembic migrations before starting the app after pulling this release.
- Existing plans keep working; reroll memory is populated only after users begin rerolling slots.
- The composed-meal maximum is enforced as a structure preference with fallback, not as an absolute hard cap when no preferred-class alternatives remain.
- If a household wants stricter long-term repetition control, add a future repeat-cooldown policy for complete meal packages rather than extending short-lived reroll memory.
