# v0.12.0 - Household users and memberships

Release date: pending tag

**Tagging gate:** do **not** create `v0.12.0` until `main` is green **and** human user testing of the full product passes. Phase 16 deferred umbrella slices may continue in parallel; decide at tag time whether the release remains `v0.12.0` or becomes `v0.13.0` if more product scope has landed.

## Summary

This release turns MealRoulette into a multi-household product. Users are global identities; households are the tenant boundary. Signup can create a household, invitations join members, household content is isolated, and Telegram delivery is linked to users with per-membership notification preferences.

Phase 15 shipped as stacked PRs rather than a single umbrella merge.

## Added

### Identity and tenancy (15A)

- UUID-backed users, households, memberships, and platform roles.
- Default-household upgrade path for existing single-tenant installs.
- Tenancy fields on auth/`/me` responses (`active_household_id`, household role, platform roles).

### Household domain scoping (15B)

- `household_id` ownership on dishes, plans, shopping, planning rules, and scheduler/Telegram household settings.
- Meal-slot reviews evolved to household- and user-scoped `meal_reviews`.
- Service and API scoping with tenant isolation coverage.

### Membership API (15C)

- Atomic household signup (user + household + household-admin membership).
- Invitation create/accept/revoke with hashed tokens and last-admin protection.
- One active household membership per user.

### Household UX (15D)

- Signup and invitation join flows in the frontend.
- Household members/invitations settings for household admins.
- Platform-admin vs household-admin settings and route boundaries.
- Personal settings reachable by ordinary authenticated members.

### Telegram tenancy (15E)

- User-level Telegram deep-link linking (QR + token).
- Household notification defaults plus per-membership subscriptions and personal reminder schedule.
- Household-scoped reminders, shopping, roulette, on-demand commands, and cooking-timer alerts.
- Legacy chat `/subscribe` deprecated for household notifications.

## Changed

- Authorization uses explicit `platform_admin` / `household_admin` / `household_member` roles instead of a single global `admin`.
- Global taxonomy and backups remain platform operations; household admins manage household content only.
- Telegram recipients require an active household membership and matching subscription toggles.

## Data and migration notes

Relevant Alembic revisions include:

- `033` — household tenancy identity foundation
- `034`–`035` — household ownership and meal reviews
- `036` — household invitations
- `037` — Telegram user links and notification subscriptions

Run `alembic upgrade head` before starting the app after pulling this release.

## Deferred (not in v0.12.0)

- Public dish catalog / publication
- Ingredient proposals
- Telegram OTP / bot login
- Password / account settings page
- User-level recipe preference API/UI
- Multi-household membership
- Household-level portable export/import

## Upgrade notes

- Existing installs migrate into one default household; rename it in Household settings if desired.
- Greenfield signup creates a new household UUID (no product “default household” for new installs).
- Household admins should invite members and have each member link Telegram from Personal → Telegram settings.
- If migration `038` aborts on duplicate Telegram identities, reconcile manually so each non-null `telegram_user_id` maps to one MealRoulette user, then re-run migrations.
