# v0.13.0 - Ingredient proposals

Release date: 2026-07-18

## Summary

This release adds the governed missing-ingredient workflow. Household users can propose ingredients that are not yet in the canonical catalog, and platform admins can resolve those proposals without letting household requests mutate global taxonomy directly.

The workflow is intentionally conservative: proposals are evidence, not catalog rows. Platform review decides whether a request maps to an existing ingredient, becomes an alias, creates a new canonical ingredient, needs more information, is rejected, or is marked duplicate.

## Added

- `ingredient_proposals` persistence with audit fields, source locale, culinary context, suggested taxonomy fields, review status, resolution type, and reviewer notes.
- Household member APIs to create proposals, list personal proposals, withdraw pending requests, and provide more information when requested.
- Platform admin APIs to review proposals:
  - map to an existing ingredient;
  - add an alias to an existing ingredient;
  - approve as a new canonical ingredient;
  - reject with a required note;
  - request more information with a required note;
  - mark duplicate with a required note.
- Proposal matching against canonical ingredients, aliases, and active proposal states.
- Ingredients-rooted frontend surfaces:
  - member proposal submission and personal proposal list;
  - platform proposal review queue and detail page;
  - breadcrumbs and navigation aligned with the Ingredients section.
- Review UI helpers for ingredient search selection, taxonomy selection, aliases, default/shopping unit choices, and conversion notes.

## Changed

- Canonical catalog mutation remains platform-admin-only; household proposals no longer imply direct catalog creation.
- Member-submitted proposal provenance is derived server-side as `manual`.
- `SECRET_KEY` is now mandatory and must not use known placeholders. Docker Compose requires a deployment-provided `SECRET_KEY`.
- Catalog ingredient and alias creation can participate in the proposal review transaction and translate uniqueness conflicts into HTTP 409 responses.

## Safety and integrity

- Migration `040` creates the Phase 16C proposal schema and aborts on unknown existing `ingredient_proposals` table shapes instead of destructively guessing.
- Known legacy umbrella-table cleanup is explicit and guarded.
- Review transitions lock proposals before catalog side effects to prevent concurrent double review.
- Catalog mutations and proposal review fields commit atomically.
- Whitespace-only proposal fields, aliases, locales, canonical/display names, and required review notes are rejected.
- Duplicate/rejected proposals do not permanently reserve canonical names; actual canonical and alias constraints remain authoritative.

- Public catalog / public dishes.
- User-level recipe ratings UI.
- Localization foundation.
