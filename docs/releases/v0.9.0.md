# v0.9.0 — Design system reconciliation and live recipe traits

Release date: 2026-07-14

## Summary

This release turns the MVP into a more coherent daily-use application. It introduces the shared UI shell and design-system primitives, cleans up catalog/planning/settings workflows, adds visual QA coverage, and fixes stale food-group classifications by computing recipe traits from current ingredients at read time.

It also includes the documentation harmonization work merged after `v0.8.0`, so the repository now has clearer boundaries between product specs, feature specs, operations runbooks, ADRs, working taxonomy files, and historical release notes.

## Added

### Design system and application shell

- Shared authenticated app shell with desktop sidebar, mobile bottom navigation, skip link, and cooking-mode full-screen behavior.
- Shared UI primitives for buttons, cards, metadata lists, disclosures, bottom sheets, status badges, segmented controls, switches, steppers, search selects, settings headers, sticky form actions, and technical values.
- Shared typography, spacing, control-size, layout, and workflow CSS tokens.
- `BrandLogo`, route navigation metadata, breadcrumbs, and page shell utilities.
- Playwright visual QA fixtures and route screenshot/overflow checks.

### Catalog and recipe experience

- Dish and recipe detail pages migrated to the shared layout system.
- Dish classification and inherited recipe metadata now use a consistent metadata presentation.
- Recipe detail now shows a food-group composition donut chart based on `food_group_weights`.
- Food-group display now derives from weighted composition instead of showing every non-zero group as equally important.
- Recipe public-key lookup endpoint:

```http
GET /api/recipes/by-key/{public_key}
```

- Public-key slug generation now keeps normal readable letters in the name portion, including letters such as `i`, `u`, `l`, and `o`; the ambiguity-safe alphabet remains limited to generated suffixes.

### Live computed traits

- Recipe and dish API responses compute traits from current recipe ingredients and ingredient taxonomy instead of trusting potentially stale `recipes.computed_traits_json`.
- Planning and scheduler paths use fresh computed traits for meal-plan items and candidate matching.
- Catalog list serialization caches reference units per list call to avoid repeated `g` / `ml` lookups.
- Significant pantry-marked ingredients now contribute to trait vectors and composition; tiny lines are still filtered by `vector_min_grams`.
- `pantry_item` is now documented as shopping/Telegram behavior, not a composition exclusion flag.

### Planning, review, and shopping polish

- Review-attention indicators are available in both desktop and mobile navigation.
- Today and review copy now handles singular/plural “needs review” grammar correctly.
- Meal status badge coverage and variants were expanded.
- Shopping item rows were extracted into a component with clearer usage-list presentation.
- Formatting helpers were added for quantities, shopping categories, breadcrumbs, dates, and review counts.

### Settings and admin workflow polish

- Settings pages use shared shells, section headers, form controls, and density rules.
- Weekly targets, scheduler, Telegram, backup, and ingredient-edit flows were migrated toward consistent form and metadata layouts.
- Overflow menus, dialogs, save-status messages, and copy-to-clipboard behavior received accessibility and interaction fixes.

### Documentation

- Added `docs/features/ui-ux-design-system.md`.
- Moved feature specs under `docs/features/`.
- Moved runbooks/process docs under `docs/operations/`.
- Added `docs/README.md` as the documentation map and naming convention.
- Archived the Phase 11 handoff under `docs/archive/`.
- Updated `README.md`, `AGENTS.md`, backlog, roadmap, MVP, ADRs, taxonomy files, and release links to reflect the harmonized documentation structure.

## Changed

- `recipes.computed_traits_json` remains in the database for compatibility/cache use, but normal API responses and scheduler/planning logic no longer treat it as authoritative.
- Trait computation now includes all recipe ingredients above `vector_min_grams`, including pantry-marked ingredients.
- Dish list, dish detail, recipe list, and recipe detail responses serialize fresh traits.
- Recipe composition uses grouped display semantics: groups at or above the display threshold remain visible, smaller groups are grouped as `Other` in chart contexts.
- The frontend route tree now runs inside the shared app shell.

## Validation

- GitHub Actions on PR #13 passed backend and frontend jobs.
- Backend: 220 tests passed.
- Frontend: 109 tests passed.
- Frontend production build passed.
- CodeRabbit review feedback was addressed and the CodeRabbit status check passed.

## Upgrade notes

- No new Alembic migration is included in this release.
- Existing recipe and dish public keys are not rewritten.
- Existing stored `recipes.computed_traits_json` values may remain stale in the database, but read APIs compute fresh traits from current recipe ingredients.
- Scheduler and planning behavior may change for recipes with significant pantry-marked ingredients, because those ingredients now contribute to vectors and food-group composition when above `vector_min_grams`.
- The frontend now includes Playwright configuration and visual QA fixtures; install frontend dependencies after pulling if local `node_modules` is stale.
